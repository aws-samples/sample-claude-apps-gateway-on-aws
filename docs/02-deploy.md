# 2. Deploy

These are the literal steps to deploy this repo into your own AWS account. Complete [01-prerequisites.md](01-prerequisites.md) first — you need an Okta client ID, issuer, and admin group name before starting.

## 2.1 Install and bootstrap

```bash
git clone <this-repo-url>
cd claude-apps-gateway-on-aws/cdk
npm install
```

If this is the first CDK deployment in this account/region, bootstrap it:

```bash
npx cdk bootstrap aws://<your-account-id>/<your-region>
```

## 2.2 Deploy all seven stacks

```bash
npx cdk deploy --all \
  -c oidcIssuer=https://your-org.okta.com \
  -c oidcClientId=0oaXXXXXXXXXXXXXXXXX \
  -c oidcClientSecret="<your Okta app's client secret>" \
  -c adminOktaGroupName=claude-gateway-admins
```

This builds and pushes two container images (the gateway and the admin console) on a temporary Linux x86_64 EC2 instance this deployment provisions for exactly that purpose — not locally on your machine — because the gateway's own Dockerfile downloads an x86_64-only binary; building it locally on an Apple Silicon Mac produces an image Fargate can't run at all. That instance is torn down automatically once the images are pushed. The full deploy order is: the network stack (VPC, subnets, NAT Gateway, security groups), the database stack (Aurora Serverless v2 Postgres), the secrets stack, the build-machine stack, the gateway stack, the admin console stack, and the VPN stack. Expect this to take 25–35 minutes — most of it Aurora cluster creation, the build machine's Docker build/push cycle, and the two ECS Express Mode services provisioning their load balancers.

The real Okta client secret is a required parameter here, passed the same way as `oidcIssuer` and `oidcClientId`, because the gateway validates its full configuration schema at boot and treats `client_secret` as required — an earlier design that deployed with an empty placeholder secret and asked you to fill it in afterward caused the gateway to crash-loop on every launch attempt, since ECS Express Mode gives up and CloudFormation rolls back the whole stack before there's ever a chance to patch it in.

**This is a reference/PoC-oriented pattern, not a production recommendation.** CDK context values (`-c`) are written verbatim to `cdk.context.json` on disk and appear in the synthesized CloudFormation template, so the secret is not protected from anyone with read access to your working directory or CloudFormation console/API in this account. For a production deployment, source the secret from an existing out-of-band Secrets Manager entry that CDK references by ARN instead of passing the plaintext value through context.

CDK will prompt you to approve IAM policy changes before creating any of the roles described in [the architecture overview](../README.md#architecture) — this is expected the first time; review and accept.

When it finishes, note the two URLs printed in the outputs:

```
ClaudeGatewayStack.GatewayEndpoint = https://cl-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx.ecs.<region>.on.aws
ClaudeGatewayAdminConsoleStack.AdminConsoleEndpoint = https://cl-yyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyy.ecs.<region>.on.aws
```

These hostnames are generated by ECS Express Mode and are not predictable before deployment — that's expected, and both stacks already patch their own `*_PUBLIC_URL` environment variable to the correct value automatically as part of the deploy (see the "URL fixer" custom resource in `gateway-stack.ts`/`admin-console-stack.ts` if you want to see how).

## 2.3 Download and connect the VPN profile

The gateway is deliberately on a private subnet with no direct internet route (see [the architecture overview](../README.md#architecture)), so reaching it — including for the sign-in step below — requires a private network path. This deployment provisions its own [AWS Client VPN](https://docs.aws.amazon.com/vpn/latest/clientvpn-admin/) endpoint for exactly this, with a ready-to-import client profile, so you don't need to set one up by hand.

1. Download the profile:

    ```bash
    aws secretsmanager get-secret-value \
      --secret-id "$(aws cloudformation describe-stacks --stack-name ClaudeGatewayVpnStack \
          --query "Stacks[0].Outputs[?OutputKey=='VpnClientProfileSecretArn'].OutputValue" --output text --region <your-region>)" \
      --query SecretString --output text --region <your-region> \
      | jq -r .ovpnProfile > claude-gateway-vpn-client.ovpn
    ```

2. Install the [AWS-provided OpenVPN client](https://docs.aws.amazon.com/vpn/latest/clientvpn-user/user-getting-started.html) if you don't already have it (any standard OpenVPN client works too — the profile uses OpenVPN's standard embedded-certificate format, with no separate cert files to manage).
3. In the client, **File → Manage Profiles → Add Profile**, and select `claude-gateway-vpn-client.ovpn`.
4. Select the profile and click **Connect**.

This endpoint uses **split-tunnel** mode, not full-tunnel — only traffic bound for this deployment's VPC goes through the VPN; everything else (including your browser's normal internet access) is unaffected. This matters later, in the sign-in step: full-tunnel mode would route your browser's redirect to Okta's public sign-in page through the tunnel too, where it isn't authorized, and the sign-in would silently hang with no visible error. If you ever see that symptom, check your VPN client is actually using the profile as generated and hasn't been reconfigured to full-tunnel.

**Do this before the sign-in step below, not after** — the admin console's sign-in flow redirects your browser to the gateway's own `/device` page, which only resolves and responds if you already have a route to its private subnet. Without the VPN connected, that page will hang indefinitely with no error at all, which can look like the deployment itself is broken when it isn't.

## 2.4 Update the Okta redirect URI

Go back to your Okta app registration ([01-prerequisites.md](01-prerequisites.md), step 3) and replace the placeholder redirect URI with the real one, using the gateway's actual endpoint from the stack output:

```
https://<your-gateway-endpoint>/oauth/callback
```

Save the change in Okta.

**If you skip this step** (or set the wrong URL), you'll get all the way through the VPN connection and the device-authorization flow, and then Okta itself will reject the sign-in with a page like this:

```
Bad Request
The 'redirect_uri' parameter must be a Login redirect URI in the client app settings: <link to your app>
Error: The 'redirect_uri' parameter must be a Login redirect URI in the client app settings...
```

That's Okta telling you exactly what's wrong — the URL above must be entered verbatim (including `/oauth/callback`) as a **Sign-in redirect URI** on your Okta app's General tab. Save the change, then start the sign-in flow again from the beginning (the device code from the failed attempt will have expired).

## 2.5 Verify the gateway booted successfully

```bash
curl -s https://<your-gateway-endpoint>/healthz
```

## 2.6 Done

At this point the gateway and admin console are both running with real Okta authentication wired up. Continue to [03-verify.md](03-verify.md) to confirm everything works end to end.
