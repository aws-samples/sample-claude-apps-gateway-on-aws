# 1. Prerequisites

Before deploying, you need:

- An AWS account with permission to create VPCs, ECS Express Mode services, Aurora clusters, IAM roles, and Secrets Manager secrets.
- [Amazon Bedrock model access](https://docs.aws.amazon.com/bedrock/latest/userguide/model-access.html) granted for the Anthropic Claude models you intend to use, in the region you deploy to.
- Node.js 20+ and the AWS CDK CLI (`npm install -g aws-cdk`) on your machine.
- Docker, running locally (CDK builds and pushes the gateway and admin console container images as part of `cdk deploy`).
- An Okta tenant where you can create an OIDC application and a group, described below.
- The AWS CLI, configured with credentials for the target account.

## Okta OIDC application

The gateway authenticates every user — developers on the CLI and admins in the console — through Okta via OIDC. You need one Okta application and one Okta group before you deploy.

1. In the Okta admin console, go to **Applications → Applications → Create App Integration**.
2. Choose **OIDC – OpenID Connect**, application type **Web Application**.
3. For the redirect URI, use a placeholder for now (e.g. `https://placeholder.invalid/oauth/callback`). The gateway's real URL is only known after it's deployed — you'll come back and update this once (see [02-deploy.md](02-deploy.md)).
4. Assign the app to whichever users or groups should be able to sign in.
5. From the app's **General** tab, note the **Client ID**. You'll pass this to `cdk deploy` as a context value.
6. Note your Okta org URL, e.g. `https://your-org.okta.com`. If you are using the default org authorization server (most tenants, unless you've explicitly created a custom one), the OIDC issuer is that bare org URL — not `https://your-org.okta.com/oauth2/default`. Confirm by fetching `https://your-org.okta.com/.well-known/openid-configuration` and checking the `issuer` field in the response.
7. Generate a **Client Secret** on the same tab. You will store this value directly into AWS Secrets Manager after deployment (see [02-deploy.md](02-deploy.md)) — it is never written to a file in this repo or passed as CDK context.

### Admin group

Create an Okta group (for example, `claude-gateway-admins`) and add to it every user who should be able to manage spend limits and model access through the admin console. Membership in this group — not network placement — is what the gateway and console use to decide who can reach admin-only functionality.

The group name is passed to `cdk deploy` as a context value (`adminOktaGroupName`) and is also the value the gateway checks against each signed-in user's `groups` claim.

### Groups claim and the org authorization server

For the gateway to see group membership, your Okta app's ID token (or `/userinfo` response) must include a `groups` claim listing each user's group memberships. Configure this under your Okta authorization server's **Claims** settings, scoping the claim to the groups you want exposed (e.g. a filter on group name, so you don't leak your entire org's group list into every token).

If you're using the default org authorization server (the common case, confirmed in this project's own deployment), it returns a thin ID token that omits `email` and `groups` regardless of claims configuration — the gateway must fall back to the `/userinfo` endpoint to get them. This repo's `gateway/gateway.yaml` template already sets `oidc.userinfo_fallback: true` for this reason. If you have a custom authorization server configured differently, you may not need this, but leaving it `true` is safe either way.

## What CDK deploy will ask for

`cdk deploy` needs four pieces of Okta configuration, all passed as [CDK context values](https://docs.aws.amazon.com/cdk/v2/guide/context.html) on the command line (see [02-deploy.md](02-deploy.md) for the exact command):

| Context key | Example value | Where it comes from |
|---|---|---|
| `oidcIssuer` | `https://your-org.okta.com` | Okta app's General tab / discovery document, step 6 above |
| `oidcClientId` | `0oaXXXXXXXXXXXXXXXXX` | Okta app's General tab, step 5 above |
| `oidcClientSecret` | (generated by Okta) | Okta app's General tab, step 7 above |
| `adminOktaGroupName` | `claude-gateway-admins` | The group you created above |

The client secret is passed as CDK context in this reference deployment for simplicity. Be aware that CDK context values are written to `cdk.context.json` on disk and appear in the synthesized CloudFormation template in plaintext — acceptable for a PoC/reference deployment, but for production use, source it from an existing out-of-band Secrets Manager entry that CDK references by ARN instead.
